# Local RAG Chatbot

A fully local Retrieval-Augmented Generation (RAG) chatbot that answers questions from your own documents — no API keys, no cloud dependency, no data leaving your machine.

Built as a learning project to understand how RAG pipelines work end-to-end: from document ingestion and chunking, through vector embeddings and similarity search, to LLM-generated answers grounded in your source material.

---

## What It Does

You drop your `.txt`, `.md`, or `.pdf` files into a folder. The system ingests them, splits them into chunks, embeds them into a vector store, and lets you query them through a web UI, terminal, or REST API. Answers are generated by a local LLM and always include the source documents they came from.

Common questions (return policy, warranty, test drives, etc.) are matched against a built-in FAQ and answered instantly without waiting for the LLM.

```
Your documents → Chunking → Embeddings → ChromaDB
                                              ↓
              User question → FAQ check → Vector search → LLM → Answer + Sources
```

---

## Tech Stack

| Component | Tool |
|-----------|------|
| Embeddings | `sentence-transformers` (`all-MiniLM-L6-v2`) |
| Vector Store | ChromaDB (file-based, persisted locally) |
| Text Splitting | LangChain `RecursiveCharacterTextSplitter` |
| PDF Parsing | `pypdf` |
| LLM | Ollama (`qwen3.5:9B`) — runs fully locally, `/no_think` mode for fast RAG |
| API | FastAPI |
| Frontend | Single-page HTML chat UI |
| Evaluation | Custom eval script with retrieval + similarity scoring |

---

## Features

- **Multi-format ingestion** — `.txt`, `.md`, and `.pdf` documents
- **Dual-mode access** — Customer mode (public info only) and Employee mode (includes confidential HR docs)
- **FAQ instant answers** — Common questions are matched and answered immediately without LLM latency
- **Conversation history** — Maintains context across follow-up questions (last 3 exchanges)
- **Web chat UI** — Clean dark-themed interface served at `http://localhost:8000`
- **Evaluation script** — 18 Q&A test cases measuring retrieval precision, answer similarity, and FAQ coverage
- **Swappable LLM** — Toggle between Ollama (local) and OpenAI via `.env`

---

## Why Local?

Running everything locally (embeddings + LLM via Ollama) means:
- No API costs or rate limits
- Documents never leave your machine — useful for sensitive or private data
- Works offline
- Full control over the model and retrieval pipeline

The tradeoff is setup overhead (pulling the Ollama model is ~6GB for `qwen3.5:9B`) and slower inference compared to a hosted API.

## Setup

### 1. Install Python dependencies

```bash
cd rag-chatbot
pip install -r requirements.txt
```

### 2. Install Ollama and pull the model

Download Ollama from [ollama.com](https://ollama.com) and install it.

```bash
ollama pull qwen3.5:9B   # ~6GB download
```

Ollama starts automatically on install, or run `ollama serve` manually.

### 3. Add your documents

Place `.txt`, `.md`, and/or `.pdf` files in the `documents/` folder. These are the files the chatbot will use to answer questions.

### 4. Ingest documents

```bash
python ingest.py
```

This reads all documents, splits them into chunks, generates embeddings, and stores everything in ChromaDB. Re-running this command clears and rebuilds the entire vector store.

### 5. Start the web UI

```bash
python api.py
```

Open `http://localhost:8000` in your browser. Use the toggle in the header to switch between Customer and Employee mode.

### 6. Or chat in the terminal

```bash
python query.py
```

Type your question and press Enter. Type `exit` to quit.

## Evaluation

Run the eval script to measure retrieval and answer quality:

```bash
python eval.py
```

It runs 18 test queries and reports:
- **Retrieval precision** — did the correct source documents get retrieved?
- **Answer similarity** — cosine similarity between expected and actual answer embeddings
- **FAQ hit rate** — how many questions were answered instantly via FAQ
- **Overall grade** (A/B/C/D)

### Model Comparison Results

I tested 4 models on the same 18-question eval set. An interesting pattern emerged — smarter models actually score lower on cosine similarity because they paraphrase and elaborate more instead of echoing the expected answer verbatim.

| Model | Intelligence | Avg Similarity | Overall Score |
|-------|:-----------:|:--------------:|:-------------:|
| Gemma 2 9B | ~20 | 0.68 | 0.84 (B) |
| Qwen3 8B (default prompt) | ~25 | 0.61 | 0.80 (B) |
| Qwen3 8B (optimized prompt) | ~25 | 0.73 | **0.86 (A)** |
| Qwen3.5 9B | **31** | 0.53 | 0.77 (B) |

**Key takeaway:** Prompt optimization mattered more than model size — the optimized Qwen3 8B prompt scored highest overall. The smarter Qwen3.5 9B gives better real-world answers but scores lower because cosine similarity penalizes rephrasing.

### Next Steps

- Optimize the system prompt specifically for Qwen3.5 9B (currently using the Qwen3 prompt)
- Replace cosine similarity with an LLM-as-judge eval (use an API model to grade answer correctness instead of embedding similarity) — this should better reflect actual answer quality and stop penalizing smarter models for paraphrasing

## How the RAG Pipeline Works

**1. Ingestion (`ingest.py`)**
Documents are loaded (txt, md, pdf) and split into overlapping chunks using LangChain's `RecursiveCharacterTextSplitter`. Chunking with overlap preserves context across chunk boundaries, which improves retrieval quality.

**2. Embedding**
Each chunk is converted into a vector using `all-MiniLM-L6-v2`, a lightweight sentence transformer model. These vectors capture semantic meaning, not just keywords.

**3. Storage**
Vectors are stored in ChromaDB, a local persistent vector database. Two separate collections are maintained — one for customer-facing documents and one that includes confidential employee documents.

**4. FAQ Check**
Before hitting the LLM, the question is checked against a list of common Q&A patterns. If matched, the answer is returned instantly.

**5. Query**
When no FAQ matches, the question is embedded using the same model. ChromaDB finds the most semantically similar document chunks via cosine similarity.

**6. Generation**
The top matching chunks are passed as context to `qwen3.5:9B` running via Ollama in `/no_think` mode (skips chain-of-thought for faster responses since the answer is already in the chunks). The LLM generates an answer grounded in those chunks and returns which source files were used.

## Configuration (.env)

| Variable | Default | Description |
|----------|---------|-------------|
| `OLLAMA_MODEL` | `qwen3.5:9B` | Ollama model name |
| `LLM_PROVIDER` | `ollama` | `ollama` or `openai` |
| `OPENAI_API_KEY` | — | Required if using OpenAI |
| `OPENAI_MODEL` | `gpt-3.5-turbo` | OpenAI model |
| `EMBEDDING_MODEL` | `all-MiniLM-L6-v2` | Sentence transformer model |
| `TOP_K` | `5` | Number of chunks retrieved per query |
| `CHUNK_SIZE` | `300` | Characters per chunk |
| `CHUNK_OVERLAP` | `50` | Overlap between chunks |

## What I Learned

- How RAG differs from pure LLM prompting — grounding answers in retrieved context reduces hallucination
- The impact of chunk size and overlap on retrieval quality
- How vector embeddings capture semantic similarity (not just keyword matching)
- Designing multi-collection access control (customer vs. employee)
- Building an evaluation pipeline to measure RAG quality objectively
- FAQ pattern matching as a practical optimization to avoid unnecessary LLM calls
- Prompt engineering matters more than model size — a well-tuned prompt on a smaller model beat a larger model with a generic prompt
- Cosine similarity is a flawed eval metric for RAG — smarter models paraphrase more, which lowers similarity scores despite giving better answers. LLM-as-judge is the better approach
