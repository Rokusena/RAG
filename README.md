# Local RAG Chatbot

A fully local Retrieval-Augmented Generation (RAG) chatbot that answers questions from your own documents — no API keys, no cloud dependency, no data leaving your machine.

Built as a learning project to understand how RAG pipelines work end-to-end: from document ingestion and chunking, through vector embeddings and similarity search, to LLM-generated answers grounded in your source material.

---

## What It Does

You drop your own `.txt` or `.md` files into a folder. The system ingests them, splits them into chunks, embeds them into a vector store, and lets you query them via a terminal or REST API. Answers are generated by a local LLM and always include the source documents they came from.

```
Your documents → Chunking → Embeddings → ChromaDB
                                              ↓
              User question → Vector search → LLM → Answer + Sources
```

---

## Tech Stack

| Component | Tool |
|-----------|------|
| Embeddings | `sentence-transformers` (`all-MiniLM-L6-v2`) |
| Vector Store | ChromaDB (file-based, persisted locally) |
| Text Splitting | LangChain `RecursiveCharacterTextSplitter` |
| LLM | Ollama (`llama3.2`) — runs fully locally |
| API | FastAPI |

---

## Why Local?

Running everything locally (embeddings + LLM via Ollama) means:
- No API costs or rate limits
- Documents never leave your machine — useful for sensitive or private data
- Works offline
- Full control over the model and retrieval pipeline

The tradeoff is setup overhead (pulling the Ollama model is ~2GB) and slower inference compared to a hosted API.

## Setup

### 1. Install Python dependencies

```bash
cd rag-chatbot
pip install -r requirements.txt
```

### 2. Install Ollama and pull the model

Download Ollama from [ollama.com](https://ollama.com) and install it.

```bash
ollama pull llama3.2   # ~2GB download
```

Ollama starts automatically on install, or run `ollama serve` manually.

### 3. Add your documents

Place `.txt` and/or `.md` files in the `documents/` folder. These are the files the chatbot will use to answer questions.

### 4. Ingest documents

```bash
python ingest.py
```

This reads all documents, splits them into chunks, generates embeddings, and stores everything in ChromaDB. Re-running this command clears and rebuilds the entire vector store.

### 5. Chat in the terminal

```bash
python query.py
```

Type your question and press Enter. Type `exit` to quit.

### 6. Or use the REST API

```bash
python api.py
```

Server starts at `http://localhost:8000`. Interactive Swagger docs at `http://localhost:8000/docs`.

**Request:**
```http
POST /ask
Content-Type: application/json

{
  "question": "What is covered in the onboarding document?"
}
```

**Response:**
```json
{
  "answer": "Based on the documents, the onboarding process covers...",
  "sources": ["onboarding.txt", "faq.md"]
}
```

## How the RAG Pipeline Works

**1. Ingestion (`ingest.py`)**
Documents are loaded and split into overlapping chunks using LangChain's `RecursiveCharacterTextSplitter`. Chunking with overlap preserves context across chunk boundaries, which improves retrieval quality.

**2. Embedding**
Each chunk is converted into a vector using `all-MiniLM-L6-v2`, a lightweight sentence transformer model. These vectors capture semantic meaning, not just keywords.

**3. Storage**
Vectors are stored in ChromaDB, a local persistent vector database. ChromaDB handles similarity search efficiently without needing a server.

**4. Query**
When you ask a question, it is embedded using the same model. ChromaDB finds the most semantically similar document chunks via cosine similarity.

**5. Generation**
The top matching chunks are passed as context to `llama3.2` running via Ollama. The LLM generates an answer grounded in those chunks and returns which source files were used.

## What I Learned

- How RAG differs from pure LLM prompting — grounding answers in retrieved context reduces hallucination
- The impact of chunk size and overlap on retrieval quality
- How vector embeddings capture semantic similarity (not just keyword matching)
- Integrating multiple components (LangChain, ChromaDB, Ollama, FastAPI) into a working pipeline
- Designing a REST API around an AI system with structured input/output
